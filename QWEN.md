# Super Tic-Tac-Toe — Project Context

## Project Overview

**Super Tic-Tac-Toe** is a Windows desktop application implementing an advanced variant of the classic Tic-Tac-Toe game. The game is played on a **9×9 grid** (81 total cells) organized as **9 smaller 3×3 sub-grids**. Players compete to win sub-grids and ultimately achieve three sub-grids in a row on the main board.

### Key Features
- **Player vs CPU** gameplay with timer-based AI moves
- **81 buttons** arranged in a 9×9 grid layout
- **Sub-grid win detection** — each 3×3 sub-grid can be won independently (highlighted in Cyan for player X, Red for CPU O)
- **Directional move routing** — where a player places their mark determines which sub-grid the opponent must play in next
- **Score tracking** — tracks player and CPU win counts across rounds
- **Built-in tutorial** — displays game rules via message box on launch

### Tech Stack
- **Language:** C# (with nullable reference types and implicit usings enabled)
- **Framework:** .NET 8.0 (Windows Desktop)
- **UI:** Windows Forms (WinForms)
- **Project SDK:** Microsoft.NET.Sdk

## Architecture & Code Organization

### Entry Point (`Program.cs`)
Standard WinForms entry point that initializes the application and launches the `SuperTC1` form.

### Main Form (`Form1.cs`)
The `SuperTC1` class contains all game logic (~1138 lines):

- **`Player` enum** — Defines X and O players
- **Button grouping system** — 81 buttons organized into 9 sub-grid groups (`buttonGroup11` through `buttonGroup33`), plus permanent reference groups (`buttonPermaGroup*`) that are never cleared
- **`InitialButtonGroups()`** — Initializes all button group lists and resets highlight flags
- **`RestartGame()`** — Resets all buttons and state for a new round
- **`CheckGame()`** — Core win detection logic; checks each sub-grid for X or O wins, applies color highlighting (Cyan for X, Red for O), and checks for overall game victory (3 sub-grids in a row)
- **`CPUmove()`** — AI logic for CPU's turn, uses a `Random` instance for move selection
- **`PlayerClickButton()`** — Event handler for player button clicks (wired in Designer)
- **`Tutorial_Text_Box()`** — Shows the rules dialog on game start
- **Win state tracking** — `playerWinCount` and `CPUWinCount` displayed in `textBox1` and `textBox2`

### UI Layout (`Form1.Designer.cs`)
Auto-generated by the WinForms Designer:
- 81 buttons (`button1` through `button81`) arranged in a 9×9 grid
- Each button is 60×60 pixels
- Sub-grids are visually separated by gaps in positioning
- Two text boxes for score display
- A `CPUtimer` for pacing CPU moves

## Game Rules (In-Game)

1. **First move:** Player X can place their mark anywhere on the board
2. **Directional rule:** The position within a sub-grid where you place your mark determines which sub-grid your opponent must play in next (e.g., placing in the top-right cell sends the opponent to the top-right sub-grid)
3. **Free move:** If sent to a sub-grid that is already full or won, the player can place their mark anywhere
4. **Win a sub-grid:** Three marks in a row within a 3×3 sub-grid (standard Tic-Tac-Toe rules)
5. **Win the game:** Three won sub-grids in a row on the main 9×9 board (horizontal, vertical, or diagonal)

## Development Conventions

- **Nullable reference types** enabled — code should handle nullability explicitly
- **Implicit usings** enabled — common namespaces are auto-imported
- **WinForms event-driven pattern** — UI events wired via `+=` handlers in Designer code
- **State management via class fields** — game state stored as instance fields on the form class
- **Console logging** — uses `Console.WriteLine()` for debug output during game checking

## Notes

- The project uses a **trial naming convention** (`Supertictactoe Trial 1`) suggesting iterative development
- The `bin` and `obj` directories are gitignored (build artifacts)
- The game targets **Windows only** due to WinForms dependency
- CPU AI uses **random move selection** rather than minimax or other strategic algorithms
